# Brainstorming

## 목적

구현 작업을 시작하기 전에 아이디어를 명확히 정의하고 디자인을 검증하는 구조화된 사고 과정

## 활성화 시점

- 사용자가 아이디어를 제시할 때
- 프로젝트 초기 계획 단계
- 요구사항 명확화 필요
- 구현 전 설계 검증 필요

## 핵심 제약

**절대 규칙**: 디자인을 제시하고 사용자의 승인을 받을 때까지 어떤 구현 작업도 하지 않음

- 코드 작성 금지
- 프로젝트 스캐폴딩 금지
- 어떤 구현 조치도 취하지 않기

## 프로세스

### 1단계: 컨텍스트 탐색

```
- 기존 파일 검토
- 프로젝트 상태 확인
- 기존 아키텍처 파악
```

### 2단계: 명확화 질문

**중요한 원칙:**

- 한 번에 하나의 질문만 (한 번에 여러 개 금지)
- 다중선택 질문 선호 (개방형 질문 최소화)
- 불필요한 기능 ruthlessly 제거
- 명확할 때까지 계속 질문

```
예시:
Q: "이 기능의 주요 목적이 무엇인가요?"
- A: 사용자 인증
- B: 데이터 관리
- C: 분석

Q: "얼마나 복잡한가요?"
- A: 간단함
- B: 중간
- C: 복잡함
```

### 3단계: 대안 제시

2-3가지 접근 방식 제시 (장단점 포함)

```
**접근 1: Client-side 렌더링**
장점: 빠른 상호작용, 단순한 배포
단점: 초기 로딩 느림, SEO 약함

**접근 2: Server-side 렌더링**
장점: SEO 우수, 보안, 초기 로딩 빠름
단점: 서버 비용, 복잡한 캐싱

**접근 3: 하이브리드 (권장)**
장점: 최적의 성능과 유연성
단점: 설정 복잡도
```

### 4단계: 디자인 제시 및 검증

섹션별로 디자인 제시:

```markdown
## 아키텍처

[세부사항]

## 데이터 흐름

[흐름도]

## 인터페이스 개요

[UI 스케치]
```

각 섹션마다 승인 요청

### 5단계: 디자인 문서화

```
docs/plans/YYYY-MM-DD-<topic>-design.md
```

승인된 내용을 마크다운 문서로 저장

### 6단계: 구현으로 전환

오직 **writing-plans** skill만 호출
다른 구현 도구는 이후에 활성화

## 핵심 체크리스트

- [ ] 한 번에 하나의 질문만
- [ ] 불필요한 기능 제거
- [ ] 2-3가지 대안 검토
- [ ] 사용자 승인 확인
- [ ] 설계 문서화
- [ ] 디자인 완료 후에만 구현

## 작은 프로젝트도 설계 필요

"너무 간단해서 설계 불필요"라는 생각 금지

**원칙**: 모든 프로젝트, 규모 관계없이 설계 단계 필수

- 간단한 설계도 존재해야 함
- 단시간에 완료 가능
